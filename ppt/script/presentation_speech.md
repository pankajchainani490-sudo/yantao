# 答辩演讲稿

各位老师好，我的项目题目是《基于机器学习的恶意流量识别系统》。本项目结合了流量分析、机器学习和网络攻防三个方向，目标是实现对正常流量、ARP 攻击、DDoS 以及木马流量的识别，并在识别之后完成告警、黑名单管理和可视化展示。

首先介绍项目背景。传统的恶意流量检测通常依赖规则和特征库，这种方式在已知攻击场景中有效，但在面对流量模式变化或新型攻击时，泛化能力有限。因此本项目尝试引入机器学习方法，从流量特征中自动学习分类边界，提高系统的灵活性和可扩展性。

本项目的总体目标有四个。第一，建立统一的数据规范，支持正常流量、ARP 攻击、DDoS 和木马流量的分类标签。第二，完成特征提取和模型训练。第三，设计一个完整的前后端系统，实现识别、告警、黑名单和回放演示。第四，考虑 Ubuntu 服务器上的部署可行性，并准备答辩展示材料。

在系统架构方面，我把整个项目分成四层。第一层是数据层，负责管理原始数据、中间数据、处理后特征数据和演示样本。第二层是机器学习层，负责特征工程、训练、评估和推理。第三层是服务层，使用 FastAPI 和 SQLite 提供接口与状态管理。第四层是展示层，使用 React 构建总览大屏、检测页、回放页、黑名单页和模型指标页。

在数据方案上，本项目采用“公共数据集加隔离实验抓包”的混合策略。公共数据集主要用来保证样本规模和可复现性，例如 CIC-IDS2017、CIC-DDoS2019 和 CTU-13。隔离实验抓包则用于满足项目对动手实践的要求，并增强答辩时的可信度。对于木马流量，我优先采用公开恶意流量样本，而不是直接运行真实木马，这样更加安全，也更适合教学和演示场景。

在特征工程方面，本项目重点提取了包长、源端口和发包频率等核心特征。同时，我又加入了一些衍生统计量，例如平均包长、最大最小包长、包长方差、每秒包数、每秒字节数、流持续时间、包长范围、每包字节数、系统端口标记以及吞吐强度。这样做的原因是，不同类型攻击在流量节奏、端口行为和包长分布上通常存在差异，这些特征能够较好地描述这种差异。

在模型选择方面，我选择了决策树和随机森林作为第一阶段基线模型。主要原因有三点。第一，这两类模型对于表格特征适配性较好。第二，训练速度快，便于快速迭代。第三，可解释性较强，能够输出特征重要性，适合课程展示和答辩说明。

在训练流程中，系统首先对样本进行特征构建，然后划分训练集和测试集，分别训练决策树和随机森林模型，并输出准确率、精确率、召回率、F1 值以及混淆矩阵。训练完成后，模型会以文件形式保存到 `ml/models` 目录中，评估报告则保存到 `ml/reports` 目录中，供后端推理和前端展示使用。

在系统实现方面，后端使用 FastAPI，提供了多个接口。最核心的是 `/predict`，它接收一条流量特征并返回分类结果、置信度、风险等级以及关键特征解释。除此之外，还提供了 `/dashboard/summary`、`/alerts`、`/blacklist`、`/replay/status`、`/metrics/summary` 和 `/settings` 等接口，用于支撑整个展示系统。数据库方面，第一阶段使用 SQLite 保存预测记录、告警记录、黑名单以及回放状态。

在预警和黑名单机制上，我并没有采用“一次恶意预测就永久封禁”的简单方式，而是设计成“模型输出加阈值规则加人工管理”的组合机制。也就是说，当同一来源多次命中恶意类别，并达到设定阈值后，系统才会自动加入黑名单。同时也保留了人工手动加入和手动删除的能力。这样既能体现自动化，又避免过度处置导致误封。

在前端展示方面，目前已经实现了多个页面。Dashboard 页面展示整体统计、攻击占比、最新告警和主要攻击源。Detection 页面可以手动输入特征并直接调用模型推理，展示识别结果和特征重要性。Blacklist 页面展示黑名单项，并支持人工增删。Replay 页面用于分阶段演示恶意流量场景。Metrics 页面展示模型指标和混淆矩阵。Settings 页面用于展示默认模型和阈值设定。

接下来是答辩展示流程。首先打开 Dashboard，让老师先看到整体统计结果。然后进入 Detection 页面，提交一条 DDoS 风格特征，展示系统如何完成识别。接着进入 Blacklist 页面，说明自动拉黑和人工管理机制。之后进入 Replay 页面，按阶段展示正常流量、ARP 攻击、DDoS 和木马场景。最后进入 Metrics 页面，解释模型结果和特征重要性。这样展示路径比较清晰，也能完整体现系统闭环。

在测试验证方面，本项目使用 pytest 作为阶段性验收机制。到目前为止，已经覆盖了项目骨架验证、数据规范验证、特征流水线验证、模型训练验证、API 接口验证以及部署就绪验证。这样做的意义在于，不仅系统功能可用，而且每个阶段都有自动化检查支持，工程化程度更高。

在部署方面，项目已经预先考虑了 Ubuntu 服务器部署可行性。目前已经补充了 nginx 配置模板、systemd 服务模板、docker-compose 模板以及部署脚本。对于答辩和课堂展示场景，我更推荐使用回放模式，而不是依赖实时抓包。这样能避免权限、网络环境和局域网可见性问题，保证展示更加稳定和可重复。

最后总结一下。本项目目前已经完成了从数据规范、特征工程、模型训练，到后端服务、前端展示、自动化测试和 Ubuntu 部署准备的完整闭环。后续如果继续扩展，我认为可以从四个方向推进：第一，接入更大规模、更真实的数据集；第二，引入更复杂的模型，例如深度学习方法；第三，增强实时抓包能力；第四，与更完整的安全运营场景进行联动。

我的汇报完毕，谢谢各位老师。
